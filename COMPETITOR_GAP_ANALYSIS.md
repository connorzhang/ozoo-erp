## 能力分层（决定能不能做）

为了避免“看起来能做、实际做不出来”，把所有功能按实现依赖分三层：

1. A 类：OZON API 可闭环（只靠官方接口 + 我们自己的数据库与流程即可完成）
2. B 类：OZON API 可部分闭环，但体验要靠自研工程化（队列/缓存/模板/重试/审核状态机/对账口径）
3. C 类：必须引入外部能力（插件采集、多平台数据、物流报价、第三方汇率/关税、市场数据）

## 功能对标差距表（竞品 → 我们的实现路径）

|---|---|---|---|---|

| 一键上架（跟卖） | 快速铺货 | 插件/采集 | 部分可做 | 如果跟卖指“基于现有商品卡创建新 offer”，需要结合 OZON 规则与可用接口；若指从前台页面采集，需插件（C 类） |
| 编辑上架（二改上架） | 控制差异化、过审 | OZON API + 内容模板 | 可做（B 类） | 类目树/属性模板 + 内容模板系统 + `/v3/product/import` + 图片导入 + 任务状态跟踪 |

| 索好评/评价管理 | 提升评分 | OZON API | 可做（A 类） | review/question/chat 全链路工作台 + 模板回复 |
| 退货处理 | 降损 | OZON API | 可做（A 类） | returns/rfbs + 对账联动 |
| 促销管理 | 提升转化 | OZON API | 可做（A/B） | actions/seller-actions + 价格保护与保本价 |
| 反查主题标签 | 选词/运营 | 不确定 | 可能需要外部 | 先用 search-queries + product-queries 近似实现；若竞品有前台标签抓取则需插件（C 类） |

## 我们能做的“强闭环 MVP”（建议）

目标：先把“上品 + 定价/利润 + 订单履约”跑通，形成可付费闭环，再做全平台情报与插件扩展。

### MVP-1（最快闭环）

- 多店铺接入：Client-Id/Api-Key 管理、`/v1/roles` 健康检查、expires_at 预警
- 上品工作台：
  - 类目树/属性模板加载（`description-category/*`）
  - 商品创建/更新（`/v3/product/import`）
  - 图片导入与状态跟踪（`/v1/product/pictures/import` 等）
  - 价格/库存批量同步（`/v1/product/import/prices`、`/v2/products/stocks`）
- 跨境定价/利润计算器：
  - 成本项模板（采购/头程/尾程/关税/平台费/包材/人工）
  - 汇率（先手动/第三方服务，后续可自动）
  - 运费：平台可选配送方式 + 试算（若受限则规则兜底）
  - 输出：建议售价、保本价、促销底价、毛利率
- 订单履约：
  - FBS/FBO/FBP 拉单、打包、标签、运单号、状态流转
  - 退货列表与处理（returns）

### MVP-2（强化卖点）

- 搜索词/选品信号工作台：
  - `search-queries/*` + `analytics/product-queries*`
  - 结合利润模型输出“建议上品清单”
- 促销与自动调价：
  - actions/seller-actions
  - pricing-strategy（竞品价、策略执行状态）
- 客服工作台：
  - chat/review/question 全链路
- 财务对账：
  - transaction list + 余额/现金流 + 订单利润回填

### MVP-3（插件与外部数据）

- 浏览器插件：
  - 采集多平台商品信息 → 标准化 → 上品模板 → 一键发布
- 外部数据池：
  - 行业爆款/类目热度/竞品监控
- 物流渠道管理：
  - 多渠道报价与时效，自动选最优（与平台可选配送方式联动）

## 差异化建议（避免同质化）

对标上品帮/毛子ERP，建议我们从 Day 1 就把差异点做出来：

- “算得准”的定价与利润：用财务流水对预估模型做后验校准，持续逼近真实利润
- “过审率”导向的上品：类目/属性模板校验 + 图片/标题/敏感词检测 + 审核状态闭环
- “多店合规”与“权限审计”：子账号权限、操作日志、关键动作二次确认
- “可扩展”的插件与数据：采集适配器、数据清洗规则、模板市场
